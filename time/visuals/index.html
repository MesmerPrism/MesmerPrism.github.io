TouchDesigner and immersive projection

The visual goal is to run the complete TouchDesigner composition with its known dependencies and controls, then deliver a stable, correctly framed image to the venue's large projection system.

Preferred route: prepared TouchDesigner workstation over NDI.

Alternative route: packaged project on the venue's experimental Linux runtime.

Approval condition: full-load test at the agreed canvas, frame rate, and latency.

Recommendation

Use a prepared workstation as the primary TouchDesigner renderer and send its final output to the venue over a dedicated wired NDI path. Promote that route to show-ready only after the actual sender, network, receiver, and projection chain sustain the required format. Test the venue Linux machine separately as an alternative path.

Three layers to keep separate

Most visual integration problems come from treating content, transport, and projection as one black box. Each layer needs its own owner, specification, and test.

1. Composition and render

The TouchDesigner project, media, fonts, plug-ins, Python packages, controls, GPU load, output TOP, and license.

2. Image transport

NDI encode, stream naming, discovery, IP and firewall setup, switch capacity, receiver compatibility, and latency.

3. Projection delivery

Native canvas, aspect ratio, slices or warping, frame rate, color space, venue ingest, operator preview, and safe fallback.

Two complete delivery routes

Both routes end at the same projection processor. The preferred route carries a finished image across the network; the alternative carries the whole project into a venue-specific runtime.

The big projection is an output, not a work surface. The visual operator should have a separate preview monitor, keyboard, and pointing device wherever possible. The public projection should receive a clean full-screen image without exposing the TouchDesigner editor or operating-system interface.

Optional audio-to-visual control

The visual system does not need to be driven by the audio system, but it can respond to loop state, tempo, levels, sound-object positions, or live analysis. Choose only the data that serves the composition.

Route selection

Route A starts with a known render environment and validates transport. Route B removes network transport but requires a complete software and hardware compatibility proof.

Decision factor Prepared workstation + NDI Venue Linux workstation
Known quantity Project environment can be frozen and rehearsed before arrival. Venue runtime is described as experimental and must be qualified on site.
Primary risk Render load, NDI encode, network headroom, receiver compatibility, and latency. Operators, plug-ins, codecs, Python, GPU drivers, file paths, controls, and display behavior.
Transport Finished image over wired Ethernet to an NDI receiver. Project runs locally and sends video through DisplayPort or HDMI.
Best role Primary after a successful full-load end-to-end test. Alternative after the packaged project passes on the exact venue image.
Shared fallback Pre-rendered representative video, a safe still or black frame, and a rehearsed switch-over procedure.

Decisions still required

These answers form the visual delivery contract and determine whether either route can be considered show-ready.

Projection canvas

  1. What are the native pixel dimensions, aspect ratio, refresh rate, and expected input frame rate?
  2. Does the venue expect one frame, a wide custom canvas, multiple slices, or a pre-warped image?
  3. What color space, range, pixel format, and scaling behavior should be used?
  4. Which device accepts the final signal, and what exact formats does it support?

NDI and network

  1. What application or hardware receives NDI, which NDI version does it support, and who configures it?
  2. Is there a dedicated wired path, and does it share switches with Dante, control, or other high-bandwidth traffic?
  3. How are discovery, IP addressing, stream naming, firewall rules, and manual source IPs handled?
  4. What sustained frame rate, missed-frame count, and end-to-end latency are acceptable?

TouchDesigner runtime

  1. Which exact TouchDesigner build and license will each machine use?
  2. What project dependencies are platform-specific or licensed separately?
  3. What exact Linux distribution, compatibility method, GPU, driver, and display server make up the venue runtime?
  4. Can the packaged project start, save, restart, and recover without exposing the editor on the public output?

Operation and A/V control

  1. Where is the visual operator positioned, and what preview and input devices are available?
  2. Which audio or spatial data should affect the visual composition?
  3. Who owns the MIDI or OSC bridge, message map, and restart procedure?
  4. What safe visual state appears if rendering, control data, NDI, or the receiver fails?

Visual validation sequence

Use the representative project, not an empty test pattern. Record measured results and approve each route independently.

  1. Lock the projection contractRecord canvas size, aspect, frame rate, color space, input format, mapping or slicing, and physical connection.
  2. Freeze the prepared workstationRecord OS, GPU driver, TouchDesigner build, license, project version, dependencies, media, fonts, codecs, and control devices.
  3. Configure the NDI pathAgree stream name, receiver, discovery method, IP and firewall settings, switch path, and who owns each component.
  4. Test at representative loadRun the heaviest real scenes at target resolution and frame rate; log sender FPS, skipped frames, network use, receiver FPS, missed frames, and latency.
  5. Verify projection framingCheck crop, scaling, warp, slices, edge blending, color, tearing, and full-screen behavior on the actual display system.
  6. Qualify the Linux alternativeOpen the exact packaged project on the venue image and test dependencies, controls, playback, output, save, restart, and crash recovery.
  7. Test A/V controlConfirm MIDI, OSC, or analysis behavior under normal operation, disconnection, restart, and stale-data conditions.
  8. Rehearse fallbackSwitch deliberately to the pre-rendered asset, safe still, or black frame and confirm the projection never exposes the desktop or editor.
