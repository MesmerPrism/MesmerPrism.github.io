TouchDesigner to venue projection Use the prepared workstation over wired NDI; qualify venue Linux as an alternative.

Render the working TouchDesigner project on a known workstation and send the finished image to the venue. This preserves the tested software, plug-ins, media, fonts, drivers, and controls.

Primary show route Wired NDI Full-load test required

Approval condition: the actual sender, network, receiver, and projection chain must sustain the agreed canvas and frame rate with acceptable latency.

Primary: prepared workstation over NDI

The show uses a known TouchDesigner environment and transports only the finished image.

USE FOR THE SHOW

Show-ready when

  • the projection canvas is confirmed;
  • the target frame rate is sustained;
  • receiver compatibility is proven;
  • end-to-end latency is acceptable;
  • a clean fallback has been rehearsed.
The large projection is an output, not a work surface.

Provide the visual operator with a separate preview monitor and controls. The public projection should never expose the TouchDesigner editor or operating-system interface.

Keep three layers separate

Each layer needs its own owner, specification, and test.

Composition and render

Project, media, plug-ins, Python, controls, GPU load, output TOP, and license.

Image transport

NDI encode, stream discovery, network capacity, receiver compatibility, and latency.

Projection delivery

Canvas, frame rate, color, mapping, operator preview, and fallback.

Preparation and alternatives

The preferred route stays visible above. Open these sections for the venue-specific deployment and validation work.

Alternative: venue Linux runtime Run the packaged project locally only after compatibility is proven. TEST FIRST

Record the exact Linux distribution, compatibility layer, TouchDesigner build, GPU, driver, display server, and license. Audit every plug-in, Python package, codec, font, movie, device input, custom operator, and file path.

Projection canvas contract Define the exact image the venue expects before optimizing scenes. 4 QUESTIONS
  1. What are the native pixel dimensions, aspect ratio, refresh rate, and expected frame rate?
  2. Does the venue expect one frame, a wide canvas, multiple slices, or a pre-warped image?
  3. What color space, range, pixel format, and scaling behavior should be used?
  4. Which device accepts the final signal, and what formats does it support?
NDI receiver and network Specify discovery, capacity, compatibility, monitoring, and latency. 6 QUESTIONS
  1. What application or hardware receives NDI?
  2. Which NDI version, pixel formats, and resolutions are supported?
  3. Is there a dedicated wired path, and what other traffic shares it?
  4. How are discovery, IP addresses, stream naming, and firewall rules handled?
  5. What sustained sender and receiver frame rates are acceptable?
  6. What missed-frame count and end-to-end latency are acceptable?

TouchDesigner runtime checklist Freeze the working environment and identify platform-specific dependencies. RUNTIME
  • Record the OS, GPU driver, TouchDesigner build, license, and project version.
  • Package media, fonts, plug-ins, Python packages, codecs, and relative paths.
  • List every device input, custom operator, and platform-specific dependency.
  • Verify startup, save, restart, full-screen output, control input, and crash recovery.
Visual test plan Approve each route independently with the representative project. 8 STEPS
  1. Lock the canvasRecord size, aspect, frame rate, color, mapping, and physical handoff.
  2. Freeze the workstationRecord OS, GPU, driver, TouchDesigner, license, project, and dependencies.
  3. Configure NDIAgree stream name, receiver, discovery, IP, firewall, switch path, and ownership.
  4. Test full loadRun the heaviest real scenes and log render FPS, skipped frames, network use, receive FPS, misses, and latency.
  5. Verify projectionCheck crop, scaling, warp, slices, blending, color, tearing, and full-screen behavior.
  6. Qualify LinuxTest the packaged project on the exact venue image, including controls, save, restart, and recovery.
  7. Test A/V controlConfirm normal, disconnected, restarted, and stale-data behavior.
  8. Rehearse fallbackSwitch to a pre-rendered asset, safe still, or black frame without exposing the editor.
