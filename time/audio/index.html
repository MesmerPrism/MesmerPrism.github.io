Looping and wave field synthesis

The audio goal is to preserve hands-on looping while making each RC-505mkII track available as a separately positionable sound object. The core design therefore separates five loop tracks before they enter the WFS renderer.

Starting point: RC-505mkII remains the loop engine.

Baseline capacity: five independent mono loop objects.

Main limitation: overdubs inside one RC track remain one combined object.

Recommendation

Assign RC Tracks 1–5 to five individual output jacks, connect those jacks to five independent interface inputs, and map them to five WFS objects. Keep loop control on the RC-505mkII and spatial control in the WFS system. Ableton may sit between the interface and renderer for recording or expansion, but it is not required for the baseline.

Track, not overdub, is the separable unit. A beat, bass line, and harmony overdubbed into the same RC track leave as one combined object. They must occupy different RC tracks—or be captured into different Ableton tracks before combining—if they need separate spatial positions.

Control ownership

The baseline is deliberately split so the performer does not need to operate the spatial system and the spatial operator does not take over loop creation.

Loop performer

Controls RC record, play, overdub, stop, undo, clear, track faders, memories, and essential RC effects.

Spatial operator

Controls the five WFS object positions, movement cues, object levels, and spatial scene recovery.

Audio system operator

Maintains channel isolation, gain structure, monitoring, optional Ableton capture, clocking, and fallback routing.

When to choose another architecture

The RC-first design is the least disruptive starting point. Use one of the other designs only when a specific artistic or capacity requirement justifies its additional control logic.

Architecture Choose it when What it preserves Main cost
RC-505mkII first Five simultaneous mono loop objects are sufficient. Full familiar RC track workflow and control. Five-object ceiling; stereo effects need a deliberate compromise.
Ableton first Independent object count and detailed automation matter more than preserving the RC workflow. A hardware control surface can preserve hands-on commands. Custom session and control design; more rehearsal and computer dependence.
RC-to-Ableton handoff More than five objects are needed over time, but new loops should still be created on the RC. Familiar creation until a loop is committed. Clocked capture, latency compensation, double-playback protection, and loss of RC control after handoff.

Stereo trade-off. Five separate RC tracks use five mono outputs. Preserving full stereo limits the basic system to three stereo feeds. A mixed layout—one stereo object plus three mono objects—is possible if it better matches the material.

Decisions still required

Resolve these before the patch is finalized. They determine object count, monitoring, stereo use, and whether Ableton is needed.

Venue and WFS

  1. What physical or network inputs does the WFS chain accept?
  2. How many simultaneous objects are available, and does one mono input map directly to one object?
  3. Where are azimuth, elevation, and distance controlled?
  4. What sample rate, buffer size, channel naming, and session format are required?

Looping practice

  1. Are distinct musical elements normally kept on different RC tracks?
  2. Which stereo delays, reverbs, panning, or other effects are essential?
  3. Are more than five independently movable objects needed simultaneously?
  4. Do completed loops need continuing RC fader, undo, overdub, or effect control?

Routing and monitoring

  1. Can SUB 2 R isolate a dry live source, or should the microphone be split before the looper?
  2. Can dry input, rhythm, master effects, and unused tracks be excluded from every object output?
  3. What low-latency monitor mix does the performer hear?
  4. Should routing be stored globally so RC memory changes cannot alter the patch?

Ableton and synchronization

  1. Is Ableton required by the venue renderer, or only desired for recording and expansion?
  2. If used, which system is clock master?
  3. How are free-length or asynchronous loops handled?
  4. What is the recovery procedure after a missed capture, double playback, drift, or computer failure?

Audio validation sequence

Approve the baseline only after every track remains isolated and the complete monitor-to-WFS path works under representative performance load.

  1. Confirm the venue contractRecord inputs, object count, renderer location, sample rate, buffer, output patch, and monitoring path.
  2. Build the five-channel patchDisable stereo linking; assign one RC track to each jack; exclude dry input, rhythm, and unwanted returns.
  3. Prove isolationRecord a unique spoken identifier on every track and confirm each downstream meter receives only its assigned signal.
  4. Test real performance useRehearse memories, faders, stop, undo, overdub, essential effects, the live microphone, and the actual monitoring position.
  5. Measure latency and gainCheck round-trip feel, feedback risk, comb filtering, noise, headroom, and the transition between live and looped sound.
  6. Add Ableton only if justifiedPrototype one capture and handoff, including clock, exact loop boundary, muting, confirmation, and failure recovery.
  7. Freeze and documentSave RC routing, venue session, channel labels, gain settings, control ownership, and an audio fallback procedure.
