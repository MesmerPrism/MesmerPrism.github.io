Rust for XR and Meta Quest experiments

Rusty XR

Rusty XR is a public Rust workspace for reusable XR tooling: data contracts, runtime configuration, Quest diagnostics, camera and depth models, BLE and LSL bridges, polar sensor helpers, SDF and particle modules, and small Quest examples. It is the source-side counterpart to the companion apps: Rusty XR owns the reusable contracts and examples; the companion tools own the operator UX, device workflows, and release handoff.

Purpose

What the core owns

The repository is not a single Quest product and not an APK distribution channel. Its job is to keep reusable Rust-side pieces public, testable, and separate from app-specific shells. That includes schemas for companion catalogs, runtime profiles, diagnostics, sensor contracts, and example code that can be exercised without mixing private application logic into the public repo.

The current public work is especially focused on Quest development boundaries: what counts as compositor passthrough, what counts as app-visible camera input, what environment depth is, and which paths are only final-display inspection through casting or MediaProjection.

Workspace crates

  • rusty-xr-contracts for shared schemas and data contracts
  • rusty-xr-runtime-config for launch and runtime-profile configuration
  • rusty-xr-quest-diagnostics for Quest device and runtime diagnostics
  • rusty-xr-camera-model and rusty-xr-depth-model for camera/depth source modelling
  • rusty-xr-ble, rusty-xr-lsl, and rusty-xr-polar for sensor and stream bridges
  • rusty-xr-sdf and rusty-xr-particles for reusable visual and simulation primitives

Public boundary

  • Reusable Rust modules and examples stay in the public core
  • Operator UX, Windows releases, and device support live in the companion-app workspace
  • Private app paths, package identities, and unreleased artifacts stay out of the public docs

Quest examples

Camera, depth, and display-source discipline

The Quest examples are small on purpose. They include contracts-only demos, a minimal Rust-native Android smoke-test APK, and a more ambitious OpenXR/Vulkan composite-layer example that can exercise synthetic, CPU diagnostic, GPU-buffer probe, and paired-camera projection tiers.

The important discipline is source separation. Native passthrough compositor layers make the room visible but do not hand the app a sampleable camera texture. Raw Camera2 or Quest camera access is a different input class. Environment depth is a runtime-generated depth surface, not raw sensor access. Scrcpy, casting, screen recording, and MediaProjection are final-display inspection paths. Rusty XR keeps those lanes separate so prototype claims do not drift.

Current example work

  • Rust-native Android smoke testing for Quest packaging and startup checks
  • OpenXR/Vulkan composite-layer APK with camera-projection shader experiments
  • Public projected-stereo modes such as screen homography and quad-surface profiles
  • Paired Camera2 GPU-buffer experiments with explicit calibration and visual-proof gates
  • Companion-driven runs that collect logs, screenshots, foreground-app state, and diagnostics

Connected projects

  • Quest Companion Tools for the Windows and Android operator surface
  • Meta Passthrough Stack for the public API/source map
  • Viscereality for the broader Quest-based research-system lineage
