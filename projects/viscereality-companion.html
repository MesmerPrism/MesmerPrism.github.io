Meta Quest operator tooling

Quest Companion Tools

Quest Companion Tools is the shared project line behind the Windows and Android utilities I build around Meta Quest work. The common problem is an outside-the-headset control surface that can install a build, launch it, cast the display, capture proof, monitor device state, and package diagnostics without asking every project to rebuild the same operator workflow.

Purpose

Why the companion layer exists

A Quest project is rarely only the APK running in the headset. Developers, researchers, and operators also need a reliable way to discover the device, move between USB and Wi-Fi ADB, install the right build, start and stop the target package, watch the final display, and record enough state to understand what happened when something fails.

This work turns that repeated friction into a reusable tooling layer. The Windows tools are the main surface today, and the Android phone lane covers the same theme for mobile operator workflows: USB-host bootstrap, Wi-Fi handoff, staged install and launch, display or screenshot proof, and diagnostics export.

Tools implemented

  • USB and Wi-Fi ADB connection management for Meta Quest devices
  • Managed discovery or installation of public tools such as adb, hzdb, and scrcpy
  • APK install, launch, stop, catalog verification, and runtime-profile handoff
  • Display casting, screenshot capture, and MediaProjection final-display frame receiving
  • Headset and controller battery readback, foreground-app checks, wake/proximity state, and process diagnostics
  • Portable diagnostics bundles for collaborators who do not have the source checkout

What it abstracts

  • Device setup that otherwise lives in terminal notes and lab memory
  • Operator actions that should be repeatable during research sessions
  • Visual proof that a build actually launched and rendered on-device
  • Release and support workflows for people outside the development repo

Current direction

Rusty XR Companion is the active public surface

The newest iteration lives in Rusty-XR-Companion-Apps. It is a Windows-first workspace with a WPF app, CLI, core library, diagnostics library, and release tooling. It is designed to sit next to Rusty XR, which owns the reusable Rust contracts, schemas, examples, and Quest-side source code.

Older utilities remain part of the lineage: the Viscereality Companion shaped the researcher-facing operator surface, the Android phone companion explored mobile ADB handoff and diagnostics, and Quest Multi Stream explored multi-device casting and visual-test capture. The public website now treats these as one companion-tooling theme rather than as separate project pitches.

Public surfaces

  • Rusty-XR-Companion-Apps for the current Windows companion, CLI, diagnostics, and release tooling
  • Rusty-XR for the reusable Rust XR crates, Quest examples, and public source taxonomy
  • Viscereality Companion docs for the earlier researcher-facing Windows operator path
  • Viscereality for the broader Quest research system that motivated part of this tooling

Boundary

  • Companion tools inspect and operate Quest workflows; they are not the headset runtime itself
  • Casting and MediaProjection are final-display inspection paths, not raw camera sources
  • Catalogs and profiles are public handoff contracts, not private lab protocol dumps
  • Individual app names are implementation history; the reusable theme is remote Quest operation and support

Use cases

What this makes easier

The practical target is a calmer Quest workflow. A contributor should be able to install a known APK, pass a runtime profile, launch the package, see the headset display, verify the foreground app, capture a screenshot, and export a diagnostic report without memorizing every command. A lab operator should be able to run the same sequence without becoming the build engineer.

The companion layer also makes mixed Rust, Unity, Android, and OpenXR work less fragile. It gives the system an external test and support surface while the headset-side code keeps evolving.

Typical workflows

  • Install a local or release-asset APK and verify that it can be launched
  • Run a camera, depth, or display-profile experiment with a recorded operator trace
  • Cast the final headset display to Windows for inspection or demonstration
  • Collect device, process, graphics, memory, and foreground-activity snapshots for bug reports
  • Hand a portable tool to a collaborator without asking them to clone the full source tree
